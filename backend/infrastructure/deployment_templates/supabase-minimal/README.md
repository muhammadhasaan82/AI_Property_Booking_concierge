# Minimal Self-Hosted Supabase

This template is for running a low-footprint Supabase stack on a small VM.

It is intentionally based on Supabase's official self-hosting flow:
- Docs: https://supabase.com/docs/guides/self-hosting/docker
- Official compose source: https://github.com/supabase/supabase/blob/master/docker/docker-compose.yml

## Goal

Keep only the services this project is most likely to need:
- `db`
- `rest`
- `auth`
- `kong`
- `meta` (optional)
- `studio` (optional)

Disable or remove the heavy services that add the most RAM and disk pressure:
- `analytics`
- `vector`
- `realtime`
- `storage`
- `imgproxy`
- `functions`
- `supavisor`
- `inbucket`

## Recommended Approach

1. Clone or copy the official Supabase `docker/` folder.
2. Delete the unneeded service blocks listed above from the official `docker-compose.yml`.
3. Remove those services from any `depends_on` sections.
4. Replace the official `volumes/api/kong.yml` with the `kong-minimal.yml` in this folder.
5. Add the `docker-compose.override.yml` in this folder when starting the stack:

```bash
docker compose -f docker-compose.yml -f docker-compose.override.yml up -d
```

6. Copy `minimal.env.example` to `.env` or merge the values into your existing Supabase `.env`.

## Optional Services

`studio` and `meta` are useful for browser-based administration, but they are not required for the booking app itself.

If you are comfortable managing Postgres with DBeaver, psql, or migrations only:
- remove `studio`
- remove `meta`
- remove the `meta` and `dashboard` routes from `kong-minimal.yml`

## Why This Helps

- Removing `analytics` and `vector` avoids the biggest disk growth path from logs and analytics tables.
- Removing `realtime`, `storage`, `imgproxy`, and `functions` cuts idle memory and image footprint.
- The override file adds Docker log rotation and smaller Postgres WAL/memory settings for small VMs.

## Important Note

If this app only needs PostgreSQL and does not truly depend on Supabase Auth or the autogenerated REST API, a plain Postgres deployment will be even smaller than a minimal Supabase stack.
